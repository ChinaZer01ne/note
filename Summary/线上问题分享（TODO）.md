## 线上性能问题如何排查
线上出现性能问题是 Java 开发者必须面对的关键挑战之一。我的排查思路是一个**分层渐进、数据驱动**的过程，核心目标是快速定位瓶颈点。以下是我的系统化排查步骤：

## 核心原则

1. **安全第一：** 优先考虑止损（如扩容、限流、重启有问题的单实例），避免问题扩大化，同时**务必保留现场证据**（堆快照、线程快照、日志等）后再进行干预。
    
2. **数据驱动：** 一切结论基于监控指标、日志、快照等客观数据，避免盲目猜测。
    
3. **分层排查：** 从宏观系统层逐步深入到微观代码层。
    
4. **闭环验证：** 任何优化措施都需要验证效果。
    

## 详细排查步骤

1. **明确现象与范围 (What & Where)**
    
    - **快速确认问题现象：** 是接口响应慢（平均RT增高、长尾）、TPS/QPS下降、错误率飙升、CPU飙升、内存溢出（OOM）、频繁Full GC、线程池满，还是整个服务不可用？
        
    - **确定影响范围：** 是整个集群、某个机房、特定实例、特定服务，还是仅某个接口/功能？是否伴随上下游服务报警？
        
    - **查看告警信息：** 第一时间检查监控系统的告警（如 CPU、内存、磁盘、网络、GC、应用指标、错误日志告警等），这能提供最直接的线索。
        
    - **关联时间点：** 问题发生的具体时间点？是否有代码发布、配置变更、流量突增（营销活动？爬虫？）、基础设施变更（网络抖动、宿主机故障）等关联事件？
        
2. **检查基础资源层 (OS & Network)**
    
    - **CPU 使用率：**
        
        - `top`/`htop`：看整体负载（Load Average），识别哪个进程/线程消耗高。`%us`（用户态）高通常是应用代码问题；`%sy`（内核态）高可能是系统调用频繁（如 IO 密集型、锁竞争激烈、上下文切换过多）；`%wa`（IO Wait）高说明磁盘 IO 是瓶颈。
            
        - `vmstat`/`mpstat`：看 CPU 分项统计、上下文切换次数 (`cs`)、中断次数 (`in`)。
            
        - `pidstat -u -p <pid> 1`：监控特定进程的 CPU 使用详情。
            
    - **内存使用：**
        
        - `free -m`：看系统整体内存使用（`used`, `free`, `buff/cache`），Swap 使用情况（过高会严重拖慢性能）。
            
        - `top`：看进程 RES（常驻内存）、VIRT（虚拟内存）大小。
            
        - `vmstat`：看 `si` (swap in), `so` (swap out)，频繁交换说明物理内存严重不足。
            
    - **磁盘 I/O：**
        
        - `iostat -x 1`：看磁盘利用率 (`%util`)、响应时间 (`await`)、读写速率 (`rkB/s`, `wkB/s`)。高 `%util` 或高 `await` 是瓶颈信号。
            
        - `iotop`：定位哪个进程在进行大量 IO。
            
    - **网络 I/O：**
        
        - `sar -n DEV 1`：看网卡带宽使用率、包量 (`rxpck/s`, `txpck/s`)、错误包/丢包。
            
        - `netstat`/`ss -s`：看网络连接状态统计（TIME_WAIT 过多？）、重传数。`netstat -natp | grep <port>` 查看特定端口的连接情况。
            
        - `tcpdump`：抓包分析（通常是最后手段，开销大）。
            
    - **系统日志：** `dmesg` / `/var/log/messages`：检查是否有 OOM Killer 活动、硬件错误、内核报错等。
        
3. **检查 JVM 层 (Java Virtual Machine)**
    
    - **GC 活动：**
        
        - **关键指标：** Full GC 频率、Young GC 频率、单次 GC 耗时（特别是 Full GC）、GC 前后各分代内存变化、晋升到老年代的对象速率。
            
        - **工具：**
            
            - `jstat -gcutil <pid> 1000`：实时查看各代使用率、GC次数和耗时。
                
            - **GC 日志：** **这是最重要的证据！** 必须确保线上环境配置了 `-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:<file-path>` 等参数。分析 GC 日志工具：`gceasy.io`, `GCViewer`。关注 Full GC 原因 (Allocation Failure, Metadata GC Threshold, System.gc, Ergonomics)、停顿时间、回收效果。
                
    - **堆内存：**
        
        - **堆快照 (Heap Dump)：** **诊断内存泄漏/对象膨胀的利器。** 在内存使用持续增长或 OOM 时触发（`jmap -dump:format=b,file=<file> <pid>` 或 `-XX:+HeapDumpOnOutOfMemoryError` 自动生成）。用 **MAT (Memory Analyzer Tool)** 或 **JProfiler** 分析：
            
            - 查找支配树中最大的对象。
                
            - 查找存在大量实例的类。
                
            - 查找因 GC Root 路径无法回收的对象（内存泄漏）。
                
    - **线程状态：**
        
        - **线程快照 (Thread Dump)：** **诊断死锁、锁竞争、线程阻塞、线程池耗尽的利器。** 连续多次（间隔 2-3 秒）获取 (`jstack <pid> > thread.dump` 或 `kill -3 <pid>`)。用工具（如 **fastthread.io**）分析：
            
            - 统计不同状态的线程数（RUNNABLE, BLOCKED, WAITING, TIMED_WAITING）。
                
            - 查找死锁（工具通常能自动识别）。
                
            - 查看大量线程阻塞在哪个锁/资源上（如数据库连接池、某个同步方法）。
                
            - 检查线程池状态（核心线程、最大线程、队列堆积情况）。
                
    - **JVM 参数与版本：** 检查 `jinfo <pid>` 或启动参数，确认内存设置 (`-Xms`, `-Xmx`, `-Xmn`, `-XX:MetaspaceSize` 等)、GC 算法选择是否合理。
        
4. **检查应用层 (Application)**
    
    - **应用监控 & Metrics：**
        
        - **微服务框架指标：** Spring Boot Actuator, Micrometer 暴露的端点（`/actuator/metrics`, `/actuator/prometheus`）。关注：HTTP 请求耗时（P50, P95, P99）、错误率、JVM 指标、数据库连接池指标（活跃连接、等待连接）、缓存命中率、自定义业务指标。
            
        - **分布式追踪：** SkyWalking, Zipkin, Jaeger。**定位慢请求的黄金标准！** 查看整个调用链路的耗时分布，精准定位是哪个服务、哪个方法、哪个 SQL 或外部调用慢了。
            
        - **日志：**
            
            - 检查应用错误日志、Warn 日志，是否有异常堆栈（如超时异常、连接池异常、序列化异常等）。
                
            - 检查关键路径的 INFO/DEBUG 日志（确保有耗时日志），分析耗时分布。
                
            - 使用 `grep`, `awk`, `sort`, `uniq` 或日志分析平台（ELK, Loki）快速聚合分析。
                
    - **数据库：**
        
        - **慢查询日志：** 这是数据库性能问题的首要切入点。分析执行慢的 SQL。
            
        - **执行计划 (EXPLAIN)：** 对慢 SQL 进行 `EXPLAIN` 分析，检查是否走错索引、全表扫描、临时表、文件排序等。
            
        - **数据库监控：** 连接数、活跃连接数、锁等待、慢查询数量、缓存命中率（InnoDB Buffer Pool Hit Rate）、磁盘 IO、CPU 使用率。工具：`SHOW PROCESSLIST`, `SHOW ENGINE INNODB STATUS`, `pt-query-digest`。
            
        - **连接池：** 检查配置（最大连接数、最小连接数、超时时间）和监控（活跃连接数、空闲连接数、等待获取连接的线程数、获取连接平均耗时）。HikariCP 的 JMX 指标非常有用。
            
    - **缓存：**
        
        - **命中率：** 低命中率可能是缓存失效策略不当或缓存穿透/击穿。
            
        - **耗时：** Redis/Memcached 操作的 P99 延迟。
            
        - **大 Key / 热 Key：** 可能造成网络拥塞或单实例压力过大。
            
        - **配置：** 连接池配置、序列化方式（避免使用 Java 原生序列化）。
            
    - **外部依赖：** RPC 调用、消息队列、第三方 API 的耗时和错误率。通过分布式追踪和 Metrics 监控。
        
5. **代码层 (Profiling) - 深入定位**
    
    - 当以上步骤将范围缩小到具体应用或模块后，使用 **Profiler** 工具进行动态分析：
        
        - **采样分析 (Sampling Profiler)：** 如 **Async Profiler**, JFR (Java Flight Recorder)。开销相对较低，适合生产环境（谨慎使用）。查看 CPU 时间或分配内存最多的方法、热点调用树。能清晰看到方法层面的 CPU 耗时或内存分配热点。
            
        - **线上诊断神器 (Arthas)：**
            
            - `dashboard`：实时监控线程、内存、GC。
                
            - `thread`：查看线程状态、堆栈，定位阻塞问题。
                
            - `trace`：追踪方法内部调用路径，输出每个节点的耗时。**定位方法内部性能问题的利器！**
                
            - `watch`：观察方法入参、返回值、异常。
                
            - `profiler`：集成 Async Profiler 进行火焰图生成。
                
        - **火焰图 (Flame Graph)：** 将采样分析的结果可视化，直观展示 CPU 时间或内存分配在方法调用栈上的分布，快速识别最宽（最耗时）的“平顶山”。
            

## 总结与优化

1. **定位根因：** 基于收集到的所有证据（指标、日志、快照、Profiling 结果），综合分析，确定导致性能问题的根本原因（可能是单一原因，也可能是多个因素叠加）。
    
2. **制定方案：** 根据根因制定优化方案。例如：
    
    - 代码 Bug：修复死循环、优化算法复杂度（O(n) -> O(log n)）、减少不必要的对象创建、优化锁范围/使用无锁数据结构。
        
    - 资源不足：垂直扩容（增大实例规格）、水平扩容（增加实例数）、优化 JVM 参数（调整堆大小、更换 GC 算法）。
        
    - 慢 SQL：优化 SQL 语句、添加合适索引、引入读写分离、分库分表。
        
    - 缓存问题：优化缓存策略、解决缓存穿透/击穿/雪崩、优化序列化。
        
    - 线程池问题：调整线程池参数（核心/最大线程数、队列类型和大小）、优化任务处理逻辑。
        
    - 外部依赖：优化调用方式（批量化、异步化）、降级/熔断、与依赖方协调优化。
        
    - 配置问题：修正错误的连接池配置、超时时间等。
        
3. **实施与验证：**
    
    - 在预发布或测试环境充分验证优化方案的有效性和正确性。
        
    - 制定灰度发布策略，逐步上线。
        
    - **严格监控优化后效果：** 对比优化前后的关键指标（RT, TPS, CPU, GC, 错误率等），确认问题是否真正解决且没有引入新问题。
        
4. **复盘与预防：**
    
    - 进行事故复盘，记录整个过程、根因、解决方案、经验教训。
        
    - **完善监控告警：** 针对本次暴露的弱点，补充关键指标监控和更灵敏的告警阈值。
        
    - **加强预案：** 制定或优化针对类似问题的应急预案（如扩容脚本、限流降级开关、一键摘流量脚本）。
        
    - **代码优化与规范：** 将优化经验沉淀为代码规范或最佳实践，避免同类问题再次发生（如避免 N+1 查询、大对象、不合理锁等）。
        

**关键点强调：**

- **监控是基石：** 没有完善的监控（基础、JVM、应用、业务），排查如同盲人摸象。必须提前建设好。
    
- **快照是证据：** GC 日志、堆 Dump、线程 Dump 是分析复杂问题的关键证据，务必在重启前获取。
    
- **工具链要熟练：** 熟练掌握 `top`, `vmstat`, `iostat`, `jstat`, `jstack`, `jmap`, `arthas`, `MAT`, 分布式追踪、日志分析等工具。
    
- **分层思维：** 由外到内（网络/OS -> JVM -> 应用 -> 代码），由宏观到微观，逐步缩小范围。
    
- **避免盲目优化：** 一定要先定位到瓶颈点，优化才能事半功倍。优化后必须验证效果。
    
- **沟通协作：** 性能问题往往涉及多个团队（运维、DBA、其他服务开发），需要有效沟通协作。
    

这就是我作为 Java 开发工程师，在面对线上性能问题时，所遵循的系统化排查思路和具体操作方法。谢谢面试官！
## Redis
### 缓存击穿

线上将字典表缓存到了Redis中，设置了`maxmemory`，并设置缓存淘汰策略为 `allkeys-lru`。结果随着Redis内存增加，造成字典库某些字段失效，造成缓存击穿，DB压力剧增，差点宕机。

#### 分析

字典库 : Redis做DB使用，要保证数据的完整性，maxmemory设置较小，采用allkeys-lru，会对没有经常访问的字典库随机淘汰 当再次访问时会缓存击穿，请求会打到DB上。

#### 解决方案

* 不设置`maxmemory`
* 使用`noenviction`策略

Redis是作为DB使用的，要保证数据的完整性，所以不能删除数据。 可以将原始数据源(XML)在系统启动时一次性加载到Redis中。Redis做主从+哨兵保证高可用。

#### 复盘
* 一般禁止将Redis做数据库使用，当然这里只是缓存。
* 热点缓存要合理处理缓存击穿问题。


> 可以举例车辆基础信息的调用，如：车辆信息过期，重建缓存未加锁，导致缓存击穿，数据库压力剧增。解决方案是通过锁来重建缓存。

> 不建议将Redis作为数据库来用。Redis宕机存在数据丢失风险。如果数据量可控的，设置了无过期时间，要保证数据库始终有完整的数据。

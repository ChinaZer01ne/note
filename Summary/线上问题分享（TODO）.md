## 线上问题如何排查

**分层次、逐步缩小范围、结合监控与日志**：

### 核心原则

1. **保持冷静，优先止血：** 线上问题影响用户，第一时间考虑是否有快速恢复手段（如回滚、重启特定实例、扩容、降级开关）。
2. **信息收集是关键：** 全面、准确地收集信息是精准定位的基础。
3. **由表及里，层层深入：** 从最外层现象（用户反馈、监控告警）开始，逐步深入到应用内部、JVM、系统、网络、基础设施。
4. **利用工具，提高效率：** 熟练使用各种监控、日志、诊断工具。
5. **沟通协作：** 及时同步信息，必要时寻求协作（运维、DBA、其他团队）。

### 详细排查步骤

#### 确认现象与影响范围 (What & Scope)

- **明确问题：** 用户反馈的具体错误是什么？（500错误？超时？功能异常？数据错误？）监控告警的具体指标是什么？（CPU 飙升？内存 OOM？Full GC 频繁？请求量暴跌？错误率飙升？RT 激增？）
- **确定范围：** 是全局性问题还是部分用户/实例/服务/区域？影响面有多大？
- **查看基础监控：**
	- **系统层：** CPU 使用率、内存使用率（特别是 Swap）、磁盘 I/O、网络流量（带宽、连接数、错误包）。
	- **应用层：** QPS、响应时间（RT）、错误率（HTTP 状态码分布，如 5xx）、线程池状态（活跃线程数、队列大小）、JVM 内存（堆/非堆）、GC 频率与耗时（Young GC, Full GC）。
	- **中间件层：** 数据库连接池状态（活跃连接、等待连接）、缓存命中率/错误率、消息队列堆积情况。
	- **业务层：** 关键业务流程成功率、核心指标波动。

#### 检查日志 (First Look - Logs)

- **应用日志：** 这是最直接的线索来源。
	- **错误日志：** 快速扫描 ERROR 级别的日志，寻找堆栈异常信息（`NullPointerException`, `OutOfMemoryError`, `TimeoutException`, 数据库死锁等）。**特别注意首次异常发生的时间点。**
	- **关键流程日志：** 查看相关业务逻辑的 INFO/WARN 日志，寻找流程中断点或耗时异常点。
	- **上下文：** 关注异常发生时的请求 ID、用户 ID、线程名、时间戳、相关参数。**聚合分析**相同异常的频率和模式。
- **GC 日志：** 如果启用了 GC 日志（强烈建议线上开启），分析：
	- Full GC 频率和耗时是否异常增长？是否发生 `java.lang.OutOfMemoryError`？
	- GC 原因是什么？（`Allocation Failure`, `Metadata GC Threshold`, `System.gc()` 等）
	- 年轻代/老年代回收前后的使用情况。
- **中间件日志：** 数据库慢查询日志、缓存客户端日志、消息队列 Broker 日志等。
        
#### 利用实时诊断工具 (Deep Dive - Profiling & Tools)

- **JVM 内置工具：**
	- `jps`： 列出 Java 进程 PID。
	- `jstack <pid>`： **最常用！** 获取 Java 进程的线程堆栈快照。**核心用途：**
		- 分析线程状态：查找大量 `BLOCKED`, `WAITING` (特别是 `TIMED_WAITING`) 状态的线程，定位锁竞争或资源等待。
		- 检测死锁：搜索 `deadlock` 关键词或分析互相等待的线程链。
		- 查看线程执行栈：了解线程正在执行什么代码。
	- `jmap -histo[:live] <pid>`： 查看堆内存中对象的直方图统计（按类、实例数、占用大小排序），**快速定位哪种对象占用了大量内存**。`-histo:live` 会触发 Full GC，谨慎使用。
	- `jmap -dump:format=b,file=heapdump.hprof <pid>`： **生成堆转储文件**。用于深度分析内存泄漏（用 MAT, VisualVM 等工具分析）。
	- `jstat -gcutil <pid> <interval> <count>`： 实时监控 GC 情况（各区利用率、GC 次数、GC 时间）。
	- `jinfo <pid>`： 查看 JVM 参数和系统属性。
- **强大的线上诊断工具 - Arthas：** **强烈推荐！** 无需修改代码或重启，动态跟踪诊断。
	- `dashboard`： 实时监控系统、JVM 线程、内存概览。
	- `thread` / `thread -b`： 查看所有线程信息 / 直接定位阻塞线程（`BLOCKED`）。
	- `watch` / `trace` / `stack`： 动态观测方法调用参数、返回值、执行耗时、调用链路。**定位慢方法、参数异常的利器。**
	- `ognl`： 动态执行表达式，查看/修改静态变量（谨慎！）。
	- `heapdump`： 类似 `jmap -dump`，生成堆转储。
	- `logger`： 动态修改日志级别（临时打开 DEBUG 日志）。
	- `profiler`： 生成火焰图，可视化 CPU 热点或内存分配热点。
- **其他 Profiling 工具：** Async Profiler (低开销 CPU/Memory 分析)、VisualVM (GUI，需连接)、JProfiler (商业，强大)。
        
#### 分析资源与依赖 (Resources & Dependencies)

- **CPU 高：**
	
	- 结合 `top -Hp <pid>` 找到占用 CPU 高的 _操作系统线程 ID_。
	- 将线程 ID 转换为 16 进制 (`printf "%x\n" <os_thread_id>`)。
	- 在 `jstack <pid>` 输出的线程堆栈中查找 `nid=0x<hex_thread_id>` 的线程，看它在执行什么代码（通常是死循环、密集计算、频繁 GC）。
	- 使用 `Arthas profiler` 生成 CPU 火焰图。
- **内存高/OOM：**
	
	- 结合 `jmap -histo` 或 `Arthas dashboard/memory` 看哪种对象占内存最多。
	- **分析堆转储 (Heap Dump)：** 使用 Eclipse MAT 或 JVisualVM。
		- 查找支配树 (Dominator Tree) 中最大的对象。
		- 查找存在大量实例的类。
		- 查找因 GC Roots 无法回收的对象（潜在内存泄漏）。
	- 检查是否有不当的缓存使用（未设置 TTL/大小限制）、大对象/集合未释放、连接（DB/HTTP）未关闭、类加载泄漏（特别是使用动态代理、OSGi 等场景）。
		
- **线程池满/拒绝：**
	
	- 检查线程池配置（核心、最大线程数、队列大小）是否合理。
	- 使用 `jstack` 或 `Arthas thread` 分析线程状态，看大量线程卡在哪里（慢 SQL？慢外部调用？锁竞争？）。
	- 检查是否有任务提交过快或单个任务耗时过长。
- **慢请求/高 RT：**
	
	- 结合链路追踪（如 SkyWalking, Zipkin）查看请求的完整调用链路，定位耗时最长的环节（应用内部？数据库？缓存？RPC？）。
	- 使用 `Arthas trace/watch` 追踪具体方法的执行耗时和参数。
	- 检查数据库慢查询（`EXPLAIN` 分析执行计划）、缓存失效/穿透/雪崩、网络延迟、同步阻塞操作（如获取锁、同步 IO）。
- **依赖故障：**
	
	- 检查下游服务（RPC/HTTP）状态、数据库连接和慢查询、缓存（Redis/Memcached）连接和命中率、消息队列堆积和消费状态。
	- 验证网络连通性（`ping`, `telnet`, `traceroute`）、防火墙规则。
#### **复现与验证 (Reproduce & Verify)：**

- 如果可能，尝试在**隔离环境**（如预发布、压测环境）复现问题，方便更深入的分析和测试修复方案。
- 基于收集到的信息和初步分析，提出可能的根因假设。
- 实施修复或优化（代码修复、配置调整、扩容、降级等）。
- **验证修复效果：** 通过监控指标、日志、用户反馈确认问题是否真正解决。

#### **总结与预防 (Summary & Prevention)：**

- 记录完整的排查过程、根因分析和解决方案。
- 思考如何避免类似问题再次发生：
	- 代码层面：优化算法、修复 Bug、资源释放、增加防御性编程。
	- 配置层面：调整 JVM 参数（堆大小、GC 策略）、线程池参数、连接池参数。
	- 架构层面：引入缓存、异步化、限流降级熔断、服务拆分。
	- 监控告警：完善监控覆盖（增加关键指标）、优化告警阈值和通知策略（避免告警疲劳）。
	- 预案演练：制定并演练回滚、扩容、降级等应急预案。
	- 容量规划：定期进行压力测试，了解系统瓶颈。


>**监控告警是眼睛：** 完善的监控是快速发现问题的基础。没有监控，排查如同盲人摸象。
>**日志是线索：** 清晰、结构化、包含足够上下文的日志是定位问题的关键证据。
**工具是效率倍增器：** 熟练掌握 `jstack`, `jmap`, `jstat`, 特别是 `Arthas` 等工具，能极大提升排查效率。
**理解 JVM 是基础：** 对 JVM 内存模型、GC 原理、线程模型有深刻理解，才能看懂工具输出的信息。
**分层思维：** 不要一开始就扎进代码里，从宏观（监控告警）到微观（代码行）逐步缩小范围。
**沟通协作：** 复杂问题往往涉及多团队（运维、DBA、网络、其他服务团队），及时沟通信息非常重要。
**优先恢复：** 在复杂问题一时难以定位时，先考虑如何最小化影响（回滚、重启、扩容），再深入排查根因。

## Redis
### 缓存击穿

线上将字典表缓存到了Redis中，设置了`maxmemory`，并设置缓存淘汰策略为 `allkeys-lru`。结果随着Redis内存增加，造成字典库某些字段失效，造成缓存击穿，DB压力剧增，差点宕机。

#### 分析

字典库 : Redis做DB使用，要保证数据的完整性，maxmemory设置较小，采用allkeys-lru，会对没有经常访问的字典库随机淘汰 当再次访问时会缓存击穿，请求会打到DB上。

#### 解决方案

* 不设置`maxmemory`
* 使用`noenviction`策略

Redis是作为DB使用的，要保证数据的完整性，所以不能删除数据。 可以将原始数据源(XML)在系统启动时一次性加载到Redis中。Redis做主从+哨兵保证高可用。

#### 复盘
* 一般禁止将Redis做数据库使用，当然这里只是缓存。
* 热点缓存要合理处理缓存击穿问题。


> 可以举例车辆基础信息的调用，如：车辆信息过期，重建缓存未加锁，导致缓存击穿，数据库压力剧增。解决方案是通过锁来重建缓存。

> 不建议将Redis作为数据库来用。Redis宕机存在数据丢失风险。如果数据量可控的，设置了无过期时间，要保证数据库始终有完整的数据。
